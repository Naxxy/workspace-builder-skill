# Client Deliverable — Context

Last updated: 2026-05-07

---

## What we are building

A repeating pipeline for producing client-facing deliverables — reports, memos, assessments, or recommendations — from an initial brief through to a polished document. Each run covers one engagement. The pipeline keeps the analysis separate from the writing so both can be reviewed independently before the final document is produced.

---

## What good looks like

- An intake summary that captures not just what the client asked for but what problem they are actually trying to solve — these are often not the same thing
- An analysis that leads with findings and recommendations, not with process description
- A deliverable that the client can read and act on without needing to ask follow-up questions — clear, structured, no unexplained jargon

---

## What to avoid

- Answering the question the client literally asked instead of the problem they need solved — flag the gap at intake if they diverge
- Burying the recommendation at the end of the analysis
- Introducing new material in the draft stage that wasn't in the analysis
- Writing for the consultant rather than the client — the deliverable is theirs, not a showcase of methodology
